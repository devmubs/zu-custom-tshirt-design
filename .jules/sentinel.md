## 2026-03-30 - [Defense in Depth] Adding Explicit Capability Checks to Administrative POST Handlers
**Vulnerability:** While administrative pages in WordPress are typically protected by the `add_menu_page` and `add_submenu_page` capability requirements, the partial files that handle POST requests (like `pricing.php`, `orders.php`, and `template-edit.php`) did not have their own explicit `current_user_can` checks.
**Learning:** In a modular plugin architecture where administrative logic is split into partial files, it's a security best practice to implement defense-in-depth by verifying user capabilities directly within the POST handling logic. This protects against potential bypasses if the admin menu structure is ever misconfigured or if these files are included in a different context.
**Prevention:** Always use `ZU_CTSD_Security::check_capability()` (or `current_user_can`) at the entry point of any administrative action that modifies data, even if the page itself is supposed to be restricted.
